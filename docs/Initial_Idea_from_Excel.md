# Initial Idea (from Excel)

本專題一開始是用 Excel 規劃差旅報支流程與資料結構，從實際報支情境出發，整理需求後再進入系統開發。

---

## 一、專題背景與目標

專題名稱：差旅與碳排管理系統  
系統類型：企業內部管理後台  
主要使用者：員工、主管  

### 想解決的問題

- 員工申報自用車里程補助時，通常需要另外查地圖估算距離，再自行計算補助金額，流程繁瑣且容易出錯。

### 目標

- 將地圖查詢、距離計算與補助金額計算整合在同一個系統中
- 減少人工計算，降低填報錯誤
- 在費用之外，補充碳排資訊，讓差旅資料更完整

---

## 二、Excel 時期的初始流程

一開始在 Excel 中的規劃偏向「月結式」報支：

- 以月份為單位建立申請單
- 員工將當月差旅資料填入同一張單
- 主管進行審核或退件
- 月底關帳後不可再修改

流程概念：

填報 → 主管審核 → 駁回 / 核准 → 關帳

---

## 三、初始功能規劃

### 1. 使用者與權限

- 區分員工與主管角色
- 員工負責填報，主管負責審核與管理部分資料

---

### 2. 差旅報支

- 一張申請單（主表）對應多筆報支明細（子表）
- 每筆明細記錄出差日期、地點與費用

---

### 3. 費用項目

初期規劃包含多種費用，例如：

- 交通費
- 住宿費
- 膳雜費
- 其他費用

---

### 4. 制度資料

- 規劃費率與相關規則（如補助金額）
- 希望能在填報時帶入計算結果

---

### 5. 延伸功能（初期構想）

- 預算與實際差異
- 通知功能（如 Email 退件通知）
- OCR 憑證辨識
- FAQ 說明
- 與會計系統串接

👉 後續開發時，優先完成核心報支流程與里程計算，上述部分未納入本次實作或僅部分保留。

其中 FAQ 的部分：

- 初期構想為以機器人方式提供問答功能
- 後續未實作機器人，改為以「問題分類 + 關鍵字搜尋」的方式提供簡單查詢

👉 以較低複雜度的方式保留基本使用說明功能

---

## 四、資料結構初始構想

Excel 階段已先拆出幾個主要資料：

- 員工資料
- 差旅申請主表
- 報支明細
- 費用項目
- 制度資料（費率）
- 客戶資料
- 預算資料
- 碳排資料
- FAQ

其中：

- 原本有設計「客戶資料」，因為一開始假設差旅主要是業務拜訪客戶  
- 後來改為全員使用後，並非所有差旅都與客戶有關，因此移除此表，讓結構更單純

## 五、系統設計演進

以下為從 Excel 規劃到實際系統開發時的主要調整：

---

### 1. 報支流程改為「明細組申請單」

原本設計為：

- 一個月一張申請單

實作後發現：

- 同一員工一個月可能有多次出差
- 若全部放在同一張單中，彈性不足

目前改為：

- 先建立每日差旅（DailyTrip）
- 每筆差旅可新增多筆費用明細
- 員工再勾選多筆差旅組成申請單（Application）

👉 流程從「月報支」改為「先建明細，再組申請單」

---

### 2. 駁回流程與狀態設計調整

在實作駁回流程時，曾遇到流程判斷變複雜的問題，主要原因包括：

- DailyTrip 與 Application 各自有狀態，造成邏輯不一致
- 曾使用「資料是否存在」來判斷是否顯示快照或允許修改
- 作廢與刪除的使用方式不明確

這些設計在流程增加後，容易出現判斷混亂與例外情況。

後續調整為：

- 由 Application 統一管理流程狀態
- DailyTrip 僅作為資料，不再有獨立簽核狀態
- 使用明確狀態欄位控制流程，例如：
  - Draft
  - Submitted
  - Approved
  - Rejected
  - Voided

同時在駁回流程中加入快照機制：

- 當主管駁回申請單時，會保存當下送審內容（快照）
- 主管查看的是駁回當下版本
- 員工則可在原資料上修改後重新送審

另外在資料處理上：

- Application 使用 Voided 表示作廢，保留申請單歷史
- DailyTrip 與費用明細在可編輯階段仍可刪除或重建

👉 透過統一狀態來源、明確欄位設計與快照機制，讓駁回、修改與顯示邏輯更清楚，也較容易維護

---

### 3. 制度資料改為版本管理

目前費率與相關制度資料：

- 不直接修改舊資料
- 改為新增新版本（依生效日）

查詢方式：

- 取「生效日 ≤ 使用日」的最新一筆

👉 避免新制度影響歷史資料

---

### 4. 作廢與刪除的使用方式

目前設計為：

- Application 使用 Voided 表示作廢，保留歷史紀錄
- DailyTrip 與費用明細在可編輯階段仍可刪除或重建

---

### 5. 地圖與距離計算分層處理

目前設計：

- 前端負責輸入與操作
- API Controller 提供地圖相關資料
- Service 負責呼叫外部地圖服務

👉 將畫面、資料 API 與外部服務分開

---

### 6. 使用者停用機制

未刪除帳號，而是：

- 使用 Identity 的 Lockout 機制停用帳號
- 搭配 SecurityStamp 讓已登入使用者失效

👉 保留資料，同時控制登入權限

---

### 7. 分析功能聚焦在費用與碳排

目前實作的分析包含：

- 總差旅費用
- 總碳排
- 員工費用排行
- 員工碳排排行

👉 與初期「預算差異」方向不同，但更貼近目前資料內容

---

## 六、整理

本專題從 Excel 的月結式報支構想出發，在實作過程中逐步調整為：

- 以每日差旅為基礎
- 由申請單統一控管流程狀態
- 搭配地圖距離計算與碳排資料
- 並加入制度版本管理與基本分析功能

整體從單純報支流程，發展為一套以「差旅資料 + 距離計算 + 碳排呈現」為核心的系統。
